## I. PURPOSE & SCOPE

This SOP defines the standards for technical documentation to ensure that it is clear, consistent, and easy to maintain. It applies to all technical documentation, including ADRs, READMEs, diagrams, and runbooks.

## II. KEY PRINCIPLES

- **Audience-Centric:** Write for the intended audience.
- **Discoverability:** Documentation should be easy to find and search.
- **Accuracy:** Documentation should be accurate and up-to-date.
- **Brevity:** Be concise and to the point.

## III. DOCUMENTATION REQUIREMENTS

- **README.md:** Every repository must have a `README.md` file that explains what the project is, how to set it up, and how to run it.
- **Architecture Decision Records (ADRs):** All significant architectural decisions must be documented in an ADR.
- **Runbooks:** All services must have a runbook that explains how to operate and troubleshoot the service.
- **Diagrams:** Use diagrams to explain complex systems and architectures. The C4 model is the preferred standard.

## IV. STEP-BY-STEP PROCEDURE

1.  **Identify Documentation Needs:** As part of the development process, identify what documentation needs to be created or updated.
2.  **Write the Documentation:** Write the documentation, following the standards outlined in this SOP.
3.  **Review the Documentation:** Have a peer review the documentation for clarity and accuracy.
4.  **Publish the Documentation:** Publish the documentation in a central location where it can be easily accessed.
5.  **Maintain the Documentation:** Keep the documentation up-to-date as the system evolves.

## V. TEMPLATES & CHECKLISTS

- **README.md Template:** A template for creating a `README.md` file.
- **ADR Template:** A template for creating an Architecture Decision Record.
- **Runbook Template:** A template for creating a runbook.

## VI. ESCALATION PATHS

- If you are unsure what documentation is required for a new feature, ask the Engineering Manager.
- If you find documentation that is out of date, you are encouraged to update it.
