## How to Use These Documents

### For New Team Members

**Week 1 Reading List:**
1. [FOUNDING_PRINCIPLES.md](FOUNDING_PRINCIPLES.md) — Understand our mission.
2. [COMPANY/VALUES.md](COMPANY/VALUES.md) — Learn how we make decisions.
3. [ENGINEERING/QUALITY_BAR.md](ENGINEERING/QUALITY_BAR.md) — Know when work is "Done."
4. [SECURITY/SECURITY_BASELINE.md](SECURITY/SECURITY_BASELINE.md) — Understand our security stance.

**Sign-off Required:** All new hires must acknowledge reading these documents.

### For Engineers

**Before Starting Work:**
1. Check [ENGINEERING/ARCHITECTURE_OVERVIEW.md](ENGINEERING/ARCHITECTURE_OVERVIEW.md) for system design.
2. Review [ENGINEERING/REPO_CONVENTIONS.md](ENGINEERING/REPO_CONVENTIONS.md) for code standards.
3. Use [AI/AGENT_TASK_PROMPT_TEMPLATE.md](AI/AGENT_TASK_PROMPT_TEMPLATE.md) when delegating to AI.

2. Confirm security compliance with [SECURITY/SECURITY_BASELINE.md](SECURITY/SECURITY_BASELINE.md).

**When Talking to Prospects:**
1. Reference [BUSINESS/SERVICE_CATALOG.md](BUSINESS/SERVICE_CATALOG.md) for capabilities.

### For Operations

**When Managing Production:**

## Updating These Documents

These documents are version-controlled and living. They evolve as we learn.

### What Requires an ADR

Changes that affect:
- Client ownership and control guarantees.
- Security baseline or compliance posture.
- Quality standards or definition of done.
- Architectural principles or technology choices.

### Change Process

1. **Draft:** Create a PR with proposed changes.
2. **Review:** Peer review by at least one senior team member.
3. **Approve:** Final approval by Dylan Thompson for significant changes.
4. **Communicate:** Notify all affected teams of the change.
5. **Update Version:** Increment document version number.
