# Scoping & Change Control (Client-Facing)
**Version:** 1.0  
**Last Updated:** 2026-03-05

## The purpose
This document prevents scope creep and protects quality.

## The rule
We commit to what is written in your Scope.

If you want more, we can do it — but it becomes a change request.

## What counts as a change
A change is anything that:
- adds a new page type,
- adds a new integration,
- changes the CMS model in a breaking way,
- adds a portal feature,
- changes the deployment mode,
- adds new content beyond the agreed content list.

## How changes work
1. You submit a change request (simple form).
2. We respond with impact (time, cost, risk).
3. You approve or reject.
4. Approved changes get added to the backlog with updated timelines.

## Why we do this
Without change control:
- timelines slip,
- budgets blow up,
- quality drops.

Change control lets you get what you want **without wrecking delivery**.
